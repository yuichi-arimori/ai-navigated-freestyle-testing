# ai-navigated-freestyle-testing

論文「AIナビゲーション型フリースタイルテストの提案」で使用した、**テストナビプロンプト**を公開するリポジトリです。

本プロンプトは、生成AIがテストの進行をナビゲートし、ソフトウェアテストの経験が浅いテスターによる、**仕様漏れ・仕様の曖昧さに起因する不具合の検出**を支援することを目的としています。

## 公開ファイル

| ファイル | 内容 |
|---|---|
| [`test-navi-prompt.md`](test-navi-prompt.md) | 生成AIへ入力するテストナビプロンプト |
| `README.md` | 本リポジトリとプロンプトの利用方法 |

`test-navi-prompt.md`には、論文の実験で使用した疑似SNS型アプリの仕様情報と、観点・ナレッジ情報をサンプルとして収録しています。

## 使い方

1. [`test-navi-prompt.md`](test-navi-prompt.md)を開きます。
2. ファイルの内容をMarkdown形式のままコピーします。（GitHubのコピー機能を利用すると、そのままコピーできます。）
3. 「仕様情報」と「観点・ナレッジ情報」を、テスト対象システムに合わせて置き換えます。
4. 編集したプロンプト全体を生成AIへ入力します。
5. 生成AIが提示した機能一覧から、テストしたい機能のNo.を入力します。
6. 提示された観点・操作例を参考にテストを実施します。
7. 別の機能、または同じ機能の新しい観点を確認する場合は、続けて機能のNo.を入力します。

## プロンプトの構成

テストナビプロンプトは、次の3つの情報で構成されています。

1. **制御ルール定義**  
   生成AIの役割、出力ルール、テストの進行手順を定義します。

2. **仕様情報**  
   テスト対象システムの既知の仕様を定義します。

3. **観点・ナレッジ情報**  
   生成AIが探索観点を作成する際に参照する、入力パターンや不具合知見を定義します。

## カスタマイズする箇所

### 制御ルール定義

通常は、そのまま利用できます。

対象システムやテスト方針に応じて、次の項目を調整できます。

- 1回に提示する観点数
- 観点の優先基準
- 出力形式
- フェーズ構成
- ユーザーへの質問方法

### 仕様情報

テスト対象システムの仕様に置き換えてください。

対象となる情報の例は次のとおりです。

- システムの前提条件
- 動作環境
- 対象機能
- 入力条件
- 操作手順
- 正常時の挙動
- エラー時の挙動
- 対象外とする操作や範囲

生成AIが、仕様に記載済みの確認と仕様未記載条件の探索を区別できるよう、既知の仕様を可能な範囲で具体的に記載してください。

### 観点・ナレッジ情報

テスト対象システムの特性や、組織で蓄積している不具合知見に合わせて追加・変更してください。

追加する情報の例は次のとおりです。

- 入力データのバリエーション
- 境界値
- 状態遷移
- 操作順序
- 同時操作
- 過去に発生した不具合
- 類似システムで発生した不具合
- 業務上重要な例外条件
- 見落としやすい仕様未記載条件

## 利用上の注意

- 生成AIの出力内容は、使用するモデルや入力内容によって変化します。
- 生成AIが提示した観点や予想される挙動が、正しいとは限りません。
- テストの実施可否、不具合かどうかの判断、影響度の評価は、対象システムを理解した人が行ってください。
- 機密情報、個人情報、認証情報などを、利用許可のない生成AIサービスへ入力しないでください。
- 本プロンプトは、仕様漏れや仕様の曖昧さに起因する不具合の検出を支援するものであり、テストの完全性や不具合が存在しないことを保証するものではありません。
- 本プロンプトを組織やプロジェクトで利用する場合は、組織の生成AI利用ルール、情報セキュリティ規程、契約条件などを確認してください。
